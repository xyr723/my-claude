# Git 规范

## Commit Message 格式

使用 [Conventional Commits](https://www.conventionalcommits.org/) 规范，中文描述：

```
<类型>(<范围>): <简短描述>

[可选的详细说明]

[可选的关联信息]
```

### 类型

| 类型       | 说明               |
|------------|-------------------|
| `feat`     | 新功能             |
| `fix`      | 修复缺陷           |
| `docs`     | 文档变更           |
| `style`    | 代码格式（不影响逻辑）|
| `refactor` | 重构（非新功能、非修复）|
| `perf`     | 性能优化           |
| `test`     | 测试相关           |
| `chore`    | 构建/工具/依赖变更  |
| `ci`       | CI 配置变更        |

### 示例

```
feat(用户模块): 添加手机号登录功能

- 支持短信验证码发送和校验
- 集成第三方短信服务

关联: #123
```

## 分支策略

- `main` / `master` — 生产分支，始终保持可部署状态
- `develop` — 开发分支（如项目采用 Git Flow）
- `feat/<功能名>` — 功能分支
- `fix/<问题描述>` — 修复分支
- `hotfix/<紧急修复>` — 线上紧急修复

## 注意事项

- **不主动执行** git commit/push，除非用户明确要求
- **禁止使用 emoji** — commit message、分支名、tag 中一律不使用 emoji
- 提交前确保代码通过 lint 和测试
- 每次提交保持原子性：一个提交只做一件事
- 禁止提交敏感信息（密钥、密码、.env 文件）
