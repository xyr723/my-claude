# 工作流规范

## 开发工作流

### 1. 理解阶段（必须）
- 先阅读现有代码，理解架构和上下文
- 确认需求边界，有歧义时主动提问
- 识别影响范围，评估变更风险

### 2. 规划阶段（复杂任务必须）
- 多文件变更、架构调整、新功能开发时，先进入 Plan Mode
- 列出变更清单和执行顺序
- 识别潜在风险点

### 3. 实现阶段
- 严格遵循 SOLID / KISS / DRY / YAGNI 原则
- 每次变更保持最小化，可独立验证
- 实现过程中持续验证，不积累问题

### 4. 验证阶段
- 代码变更后运行相关测试
- 检查 lint/type-check 是否通过
- 确认无引入新的警告或错误

## 问题排查流程

1. **复现** — 确认问题现象和触发条件
2. **定位** — 使用工具搜索和追踪，基于事实而非猜测
3. **分析** — 理解根因，而非只看表象
4. **修复** — 最小化修改，不引入副作用
5. **验证** — 确认修复有效，回归测试通过

## 代码审查检查清单

- [ ] 是否符合项目既有架构模式？
- [ ] 是否有重复代码可以抽取？
- [ ] 是否有未使用的变量/导入？
- [ ] 错误处理是否完备？
- [ ] 命名是否清晰且一致？
- [ ] 是否需要补充测试？
- [ ] 是否有性能隐患（不必要的重渲染、内存泄漏等）？

## 工具使用优先级

| 场景         | 优先工具                     |
|--------------|----------------------------|
| 文件搜索     | Glob                        |
| 内容搜索     | Grep（ripgrep）             |
| 读取文件     | Read                        |
| 编辑文件     | Edit                        |
| 创建文件     | Write                       |
| 代码探索     | Task(Explore)               |
| 复杂任务     | Task(Plan) → 实现           |
| 终端操作     | Bash                        |

## 安全红线

- **绝不**在代码中硬编码密钥、令牌、密码
- **绝不**未经确认删除文件或执行破坏性操作
- **绝不**主动执行 git commit/push（除非明确要求）
- **绝不**修改不在当前任务范围内的代码
- **绝不**向生产环境发送未经确认的请求
